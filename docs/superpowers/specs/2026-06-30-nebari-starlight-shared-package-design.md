# Design: `@nebari/starlight` shared Starlight theme package

- Date: 2026-06-30
- Repo: `nebari-dev/starlight`
- Status: design approved, pending spec review
- Author: Chuck McAndrew

## Context

Nebari software packs each ship their own documentation. Today the docs
ecosystem is built on Hugo: each pack deploys an independent Hugo site that
consumes the `nebari-hugo-theme` module, and a Cloudflare Worker
(`software-pack-dashboard/worker`) routes `packs.nebari.dev/<slug>/` to the
right per-pack Cloudflare Pages host. The central dashboard at
`packs.nebari.dev` is itself a Hugo site generated by
`software-pack-dashboard/generate.py`.

We are moving the docs ecosystem from Hugo to Astro + Starlight, themed with
`nebari-design` (the Nebari shadcn component registry and brand-token source).
This spec covers the **first deliverable**: a shared Starlight theme package,
published from the new `nebari-dev/starlight` repo, that every Nebari docs site
will consume.

## Ecosystem goals

The migration as a whole must satisfy eight goals:

1. Each pack maintainer only worries about their own docs.
2. Limit the blast radius of a maintainer's docs change.
3. Separate presentation from content: maintainers write only markdown.
4. A unified presentation of software packs for end users.
5. Unified search across all pack docs.
6. A single entry point to pack docs.
7. Maintainers keep PR preview deploys for docs changes.
8. Versioned docs by release.

## Architecture decision: federated, not centralized

Starlight is a single-site static generator. The naive way to get unified
search and presentation (goals 4, 5, 6) is to pull every pack's docs into one
Astro build, but that breaks goals 1, 2, and 7: a monolithic build means one
maintainer's broken markdown fails the whole site, PR previews (which are
per-repo on Cloudflare Pages) have nothing to render, and the deploy lifecycle
leaves the maintainer's hands.

We keep the **federated** model (each pack deploys independently, the Worker
routes subpaths) and deliver the "unified" goals without centralizing:

- **Unified search (goal 5):** Starlight's default search is Pagefind, which
  supports multisite search by merging indexes at runtime in the browser via
  `mergeIndex`. Because the Worker already proxies every pack under
  `packs.nebari.dev/<slug>/`, all Pagefind bundles are same-origin to the
  central site, so we avoid the CORS configuration multisite search normally
  needs.
- **Unified presentation (goal 4) + markdown-only (goal 3):** instead of the
  `nebari-hugo-theme` Hugo module, every site consumes the shared Starlight
  plugin defined in this spec.

The migration decomposes into three sub-projects, each with its own
spec/plan/build cycle:

- **A. Shared Starlight theme package** (this spec) - the foundation everything
  depends on.
- **B. Central dashboard** (`software-pack-dashboard`) - landing page + the
  auto-generated pack table converted to Astro + Starlight, hosting the unified
  multisite search. Worker, `generate.py`, and `generate_routes.py` updated.
- **C. Per-pack docs template** - the Starlight replacement for
  `nebari-hugo-theme` usage, with PR-preview and versioning config; piloted on
  one pack, then rolled out.

## This package: `@nebari/starlight`

### Purpose

A reusable Starlight plugin that gives every Nebari docs site (the central
dashboard and each software pack) one unified Nebari look and shared docs
behavior, so maintainers write only markdown plus a tiny config. Delivers goals
3 and 4; built so it does not preclude 5 or 8.

### Distribution and consumption

- Published as the npm package **`@nebari/starlight`** from this repo.
- Consumed as a Starlight plugin:

  ```js
  // a pack's astro.config.mjs
  import starlight from '@astrojs/starlight';
  import { nebari } from '@nebari/starlight';

  export default defineConfig({
    integrations: [
      starlight({
        title: 'Superset Pack',
        plugins: [nebari()],
        // pack-specific: sidebar, base, social overrides, etc.
      }),
    ],
  });
  ```

- **EffVer** versioning (`vMACRO.MESO.MICRO`, per the Nebari convention).
  Consumers pin to a macro version so a presentation change cannot silently
  break a pack. This is how blast-radius isolation (goal 2) is honored at the
  presentation layer.
- Build tool: **Bun**, matching `nebari-design`'s toolchain.

### Integration approach: tokens + native Astro components

`nebari-design` ships React/shadcn/Tailwind-v4 components plus OKLCH design
tokens; Starlight is Astro-native. We consume **only** the OKLCH tokens, fonts,
and logos and build the Starlight chrome as native Astro component overrides.
No React is shipped. This mirrors exactly what `nebari-hugo-theme` does today
(it consumes only the color tokens), keeps the docs sites fast, and avoids
pulling shadcn/Tailwind-v4 build complexity into every consumer. Most shadcn UI
components are not needed for docs chrome.

### Token sourcing: vendored, by precedent

`@nebari/theme` is a shadcn registry item (`npx shadcn add @nebari/theme`), not
a conventional importable npm package. `nebari-hugo-theme` already vendors the
OKLCH tokens inline (copied from `@nebari/theme`'s `globals.css` at theme-pin
time, with a provenance header), specifically to avoid a build step or network
fetch. We mirror that proven pattern:

- `src/styles/nebari-tokens.css` holds the vendored OKLCH tokens with a
  provenance header.
- `scripts/sync-tokens` refreshes them from `@nebari/theme`.
- A separate mapping layer translates the shadcn-style token names
  (`--primary`, `--background`, ...) onto Starlight's variables.

### What the plugin provides

1. **Theme CSS (core).** Vendored OKLCH tokens plus a mapping layer translating
   them onto Starlight's variables (`--sl-color-accent*`, `--sl-color-gray*`,
   text/background) for light and dark, injected via `customCss`.
2. **Fonts.** Self-hosted Poppins (headings), Atkinson Hyperlegible (body),
   Fira Code (code), wired to `--sl-font` / `--sl-font-mono`. Same woff2 files
   the Hugo theme self-hosts; no CDN fetch at runtime.
3. **Logos and favicon.** `nebari-design` light/dark logo-mark via `logo.src`.
4. **Config defaults** via `updateConfig()`: GitHub social link, edit-link
   base, favicon, and sensible Starlight defaults, all overridable by the
   consumer.
5. **Minimal component overrides.** Start with a branded `Footer` only (and
   `SiteTitle` if the logo requires it). Add more overrides only when the brand
   actually diverges from a Starlight default.
6. **Subpath docs.** Document `base: '/<slug>/'` so a pack served at
   `packs.nebari.dev/<slug>/` links correctly behind the Worker, and verify the
   theme's own assets resolve under a non-root base.

### Repo layout

```
src/
  index.ts                  # the plugin: { name, hooks } -> config:setup
  styles/nebari-tokens.css  # vendored OKLCH tokens (provenance header)
  styles/starlight.css      # token -> Starlight mapping + typography
  fonts/                    # self-hosted woff2 + font-face.css
  assets/                   # logos, favicon
  components/Footer.astro   # minimal branded overrides
scripts/sync-tokens.*       # refresh vendored tokens from @nebari/theme
docs/                       # a Starlight demo site that consumes the plugin:
                            #   dev harness + visual reference + the package's
                            #   own docs (dogfood; itself the first consumer)
README.md
```

The Starlight plugin API supports this shape: a plugin is an object with `name`
and `hooks`; its `config:setup` hook calls `updateConfig()` (logo, social
links, defaults), `addIntegration()`, and can register `customCss`.

## Out of scope (deferred)

- **Multisite Pagefind `mergeIndex`** (goal 5): hub-only, additive runtime
  browser config, deferred to the dashboard spec (B). The package ships only
  standard themed Pagefind search.
- **Auto-generated pack table / `generate.py`**: dashboard (B).
- **Per-pack versioning UX** (e.g. `starlight-versions`): per-pack template (C).
  The package may later add a recommended versioning recipe.

## Testing strategy

A theme package has light unit-test needs; verification is mostly build-time and
visual. Planned coverage:

- **Build smoke test** (automated): demo builds with exit 0, compiled CSS
  contains the mapped Starlight variables for light and dark, no external font
  hosts, Pagefind index emitted. Covers journeys 2, 4, 6, 7.
- **Playwright + axe** (automated): theme toggle, search query, and
  accessibility checks on the built demo. Covers journeys 3, 7, 10.
- **Base-path build** (automated): a second build with a non-root `base` plus
  URL assertions. Covers journey 9.
- **Narrated checks**: screenshots and transcripts for visual branding, token
  sync, and the EffVer tag. Covers journeys 1, 5, 11, 12.

## Risks and open questions

- **`@nebari` npm scope availability.** Publishing `@nebari/starlight` assumes
  the `@nebari` npm org/scope is owned by the team. If not, fall back to
  `@nebari-dev/starlight-theme` or an unscoped name. Resolve before first
  publish; does not block local development against a git install.
- **Token name drift.** The mapping layer depends on `@nebari/theme`'s shadcn
  token names. If upstream renames tokens, `sync-tokens` plus the mapping layer
  must be updated together. The provenance header records the pinned source.
- **Starlight version compatibility.** Pin the Starlight peer dependency and
  document the supported range; component overrides are the most likely thing to
  break across Starlight majors.
